# Decision-Making and Failure-Mode Design for a Local Model-Orchestration Kernel (MoK)

## Router Feature Engineering  
Effective routers use signals beyond raw prompt text.  In practice, they include input metadata (e.g. **context length**, modality flags indicating text vs. code vs. vision), and **historical performance signals** for each expert. For example, one system built a simple classifier (via TF-IDF) to identify task category (coding, math, etc.) and then scored each model by combining its past accuracy on similar queries, latency history, and even user feedback.  Concretely, models were ranked by a weighted sum: *0.4×benchmark_score + 0.3×memory_score + 0.15×latency_score + 0.15×feedback_score*.  The “memory score” came from storing each past query + chosen model + outcome in a vector database, so that similar inputs boost models that succeeded before.  Domain-specific features can also help – e.g. counting the number of variables or operations in a math problem.  In summary, MoK routers should consider **prompt features** (length, syntax, multimodality), **system state** (which experts are loaded or recently failed), and **performance history** (per-expert latencies, error rates, and user satisfaction) when featurizing a query.

## Router Abstention and Escalation Policy  
Routers must know when they are “not sure” and should defer or escalate.  A simple pattern is **confidence thresholds**: if the router’s best score or classification confidence is below a cutoff, it declines to route normally and instead hands off to a fallback.  For example, the LocalForge pipeline uses a two-stage route: if the lightweight classifier’s confidence <0.6 it falls back to a generalist model, and if <0.3 it even routes to *any* expert that had ≥60% success on similar past queries.  More principled schemes use **uncertainty-based gating**.  Recent work suggests decomposing model uncertainty into “reducible” vs “irreducible” parts: if uncertainty is irreducible (intrinsic ambiguity), the router should *abstain* or escalate to a safer process; if it is reducible (missing data), it can query stronger experts.  In MoK this might mean: **on low confidence or high uncertainty, route the query to a coordinator (a planning module) or multiple experts**, or default to a robust fallback expert.  In extreme cases (e.g. repeated failures), escalate to a human or a higher-level planner.  The key is to include an “I’m not sure” branch in the routing logic and define clear triggers (confidence score, too many chained failures, etc.) that switch control to an escalation path.

## Multi-Expert Routing Strategy  
Deciding *how many* experts to invoke is a key policy choice.  A router might choose a single best expert, or it can schedule multiple experts either in parallel (ensemble) or in sequence (chain-of-thought).  One pattern is **cascading (sequential chaining)**: start with a cheaper model, and only if its answer seems inadequate (low confidence or format error) call a stronger one.  Indeed, the xRouter system is built as a “tool-calling” router that can **answer directly or invoke one or more external models** as needed.  Another pattern is **parallel ensemble**: dispatch the query to several experts concurrently and then merge their outputs.  Consensus-based approaches in Chain-of-Experts (CoE) architectures do this: they send inputs through multiple specialized models and then aggregate results by voting or averaging.  For example, CoE frameworks may route tokens sequentially through a series of experts (each refining the answer), or use an ensemble vote among specialist outputs.  In practice, MoK could for instance run two experts in parallel if the router is uncertain, then use a merge policy (below) to pick or combine answers.  The trade-offs are clear: multiple experts cost more latency but can boost reliability.  The router’s policy might allow: “Use one expert by default; if confidence is low, run two experts in parallel; if still unresolved, try a sequential chain of experts.”

## Coordinator Merge Strategies  
Once multiple experts have produced outputs, the system must merge or select among them.  Possible strategies include:

- **Pick-One (Best)**: Choose the single highest-scoring answer (by some heuristic). For instance, the EnsLLM code-ensemble approach generates many candidate programs and then picks the one that best aligns with consensus (using metrics like CodeBLEU similarity and execution equivalence).  In general, this is akin to picking the highest-confidence output among experts.

- **Majority Vote/Consensus**: Have each expert “vote” on the answer. In classification tasks, a simple majority-rule can be used. CoE surveys note using majority-voting ensembles and weighted voting schemes to boost accuracy.

- **Synthesis/Refinement**: Use a strong aggregator to merge outputs. For example, one could prompt an LLM to *synthesize* a final answer given all experts’ outputs. Iterative Consensus Ensembles (ICE) go further: LLMs iteratively critique and refine a joint answer through rounds of feedback.

- **Verify-then-Merge**: Apply automated checks or a “judge” model. E.g. for code, you might test each expert’s answer and choose the one that passes validation. For text, a verifier model or rule engine can score each response’s correctness before merging. A weak form is to discard answers that fail basic format/logic checks.

In practice, MoK’s coordinator might implement a multi-stage policy: e.g. generate answers from N experts, then either pick the majority/aggregate (like a vote) or send all answers to a “refinement expert” that tries to merge them.  The design should consider cost of the merge step vs benefits: a neural synthesis step adds overhead but can catch errors.

## Failure Recovery Design  
Robust orchestration requires handling errors gracefully.  Key failure modes and mitigation patterns:

- **Timeouts/No Response**: If an expert call times out or stalls, the router can *retry* or switch to an alternate model. The SHIELDA taxonomy suggests “External Call Timeout: Fallback handler” (e.g. skip or abort current step) and possibly retry an alternate API. In practice, MoK might try the next-best expert or escalate to a simpler plan on timeout. Critical timeouts could also **escalate** to a human or supervisor process if repeated.

- **Malformed Outputs**: If an expert returns unparsable or incorrectly formatted output (e.g. invalid JSON), use a safe default. The SHIELDA guideline “Fallback Output: default template if structure is broken” applies. For example, discard that output and either re-call the model or ask another expert. It’s crucial to sanitize outputs and catch parsing errors to prevent pipeline crashes.

- **Low-Confidence Answers**: If an expert responds but with low internal confidence (or contradicts known facts), the coordinator can either reject that answer or mark it for fallback. For instance, if two experts contradict, MoK may choose the majority opinion or escalate to a third expert to break the tie.

- **OOM / Resource Failures**: If an expert throws an out-of-memory or resource error, the system should evict that model, free resources, and possibly retry on lighter model. Metrics on such errors should degrade that expert’s health (see below). A simple policy: **penalize costly models** that OOM frequently, and do not retry them immediately on critical tasks.

- **Tool Invocation Failures**: If MoK uses tools or API calls, handle API errors by switching endpoints or using cached results. For example, have a fallback tool or static answer if a search API fails. SHIELDA notes using retry logic for UI/API failures.

- **Contradictory Expert Outputs**: If multiple experts disagree, strategies include: ask a verifier (like an AI “referee”) to choose; or present all answers to the user; or fall back to majority. This ties into the merge strategy above.

In all cases, incorporate a **fallback control** table. For example, Table 3 in SHIELDA shows patterns like: on external timeout, either skip that call or abort the workflow and compensate later. MoK should define similar policies: which step to skip, whether to abort entirely, or to continue with degraded quality. It’s often safest to abort or escalate if critical steps fail repeatedly.

## Cold-Start Policy  
Before MoK has collected enough routing data, it needs a rule-based policy.  A common fallback is to route **all queries to the most capable expert** by default (e.g. the largest model) or use simple heuristics.  In fact, recent work shows that a heuristic “Adaptive LLM” (always use a stronger model when in doubt) can outperform untrained routers under domain shift. In practice, one might start by always using a generalist model or one matched by prompt keywords until the router gains data. The LocalForge example uses initial thresholds: if classification fails, it picks any model with a history of success on similar tasks. Similarly, MoK could use **ε-greedy exploration**: with small probability, try a random expert to gather data. Also consider **synthetic data**: as in TRouter, generate example queries across task categories to pre-train the router (though this risks bias). The key is to avoid “cold confusion” by using safe defaults (e.g. all to GPT-4 for now) and slowly accruing logged outcomes for learning.

## Expert Registry Schema Design  
Each expert (model, adapter, or tool) should carry rich metadata.  At minimum, record its **identity and capabilities**: a name, its **role/domain** (e.g. “code generator”, “legal QA”), and its **base family/version** (e.g. Llama-2-70B, GPT-4o).  Include **modality** (text, vision, multimodal) and context length limit.  Also track **performance metrics**: e.g. benchmark scores or specialty proficiencies.  Store a **cost model** (e.g. tokens/sec, expected GPU load) to estimate latency and expense.  **Health state** fields should monitor current status (loaded in memory? recently crashed? error rate) and a running **trust/reliability weight**. A Bayesian router example updates a “reliability weight” for each agent based on its observed evidence – MoK can mirror that by lowering the trust score of an expert as it fails or by raising it when it consistently performs well.  Finally, keep **versioning and lineage**: the training data or fine-tuning history for adapters, so one can trace any quality/regression changes. In summary, the expert registry is like an ML model catalog: include *model_name, version, adapter_fingerprint, domain_tags, modality, context_limit, cost_parameters, current_load, error_rate, trust_score*, etc. This structured metadata will support routing decisions and monitoring.

## Route Schema Design  
The MoK route record (“mok.route.v1”) must be carefully designed for replay and learning. Each log entry should capture the **full routing decision** and context. At a minimum include: a unique request ID, timestamp, the user prompt and context state, and all router inputs (features). Crucially, record which expert(s) were called and **the output(s)** they produced, along with relevant costs (latency, token count, memory swaps). Also log any router confidence scores or reasons (e.g. “low confidence -> fallback”). Finally, include the **evaluation of the result**: whether it succeeded or any quality metric (such as user rating or automated check). This observational log mirrors what Tsiourvas et al call a dataset with only the outcome of the deployed model. Having such schema means each example is self-contained (input, route taken, output, cost, outcome). It should be stable and versioned so that as MoK evolves, old data remain interpretable. This design enables both offline training of new routers and replay analysis: one can replay the same prompts, simulate alternate routes, and attribute rewards properly, so long as all needed fields are recorded.

## Session Memory Policy  
MoK should maintain a **session memory** that persists across user turns. Relevant state elements include: **summaries** of the conversation or documents, the **history of routes** taken (which experts answered which questions), any **tool results or retrieved documents**, and the **failure/retry flags** from previous turns. For example, if a RAG (retrieval) step was performed, the retrieved knowledge snippet could be cached. If an expert was loaded or preferred earlier, that could inform sticky load. The persistent memory might also store the evolving user profile or instructions. In general, all context necessary for continuity should be kept in memory: e.g. if we used a slow specialty model on a prior turn, note that so the router can prefer it (resident expert) in future. Benchmarks like Momento emphasize keeping long-term memory to support multi-session consistency. Concretely, MoK could write each turn’s summary and route into a memory store, and retrieve it at the start of the next turn. This ensures the router and experts see the full conversation flow, and avoids forgetting that “we already attempted this with Model X”.

## Replay and Simulation Design  
Offline replay is critical for testing routing changes. A simple approach: **log-replay**. Replay historical logs (requests and any outcomes) through new router implementations, simulating what would happen. However, a naive replay (running live queries through models) is costly. Instead, MoK should simulate expert responses using existing logs or proxies. For reliability, ensure replay is **on recorded prompts with actual model outputs** (not synthetic). One method is to freeze the expert outputs from logs (treat them as “ground truth”) and evaluate routing decisions against those. More advanced is to use causal inference: treat logged data (one model’s output per query) as **observational data** and apply doubly-robust estimators to infer counterfactual outcomes. In any case, avoid “simulator fiction” by validating the replay on held-out real interactions. For eviction or routing experiments, consider **A/B testing** in shadow mode (see below) to complement replay.

## Counterfactual Data Collection  
Evaluating “what other experts would have done” is challenging without exponential cost. One strategy is **speculative execution**: for a random subset of queries, actually run the query on multiple experts (parallel calls) and log all outputs. This gives real counterfactual data at extra inference cost but only on sampled queries. Another strategy is **bandit exploration**: occasionally route a query to a less-likely model (with some small probability) to gather its response, and use off-policy evaluation (IPS weighting) to estimate performance. The causal routing framework highlights using **observational logs with counterfactual estimators**. That is, if only Model A was used historically, one can train a model to predict what Model B would have done (e.g. via supervised learning on queries), then estimate alternate outcomes. In practice, combine cheap heuristics (e.g. a smaller model that mimics a larger one) to approximate missing outputs. Also, as [11] suggests for memory lookups, batch processing can amortize cost: embed the input once and retrieve similar past queries for all experts together. Overall, MoK should log as much as possible and selectively collect multi-expert answers, then apply offline estimators to fill gaps.

## Reward Design  
For learned routing (e.g. RL or bandits), define a clear reward that balances multiple factors: quality, latency, and resource costs. A common formula is *Reward = QualityScore – λ×Cost*. For example, xRouter’s reward gives **+1 for task success, 0 if it fails, and subtracts a cost penalty** proportional to model usage. In their words: “no success, no reward; on success, cheaper is better”. Concretely, you might define quality as correctness (1/0) or scaled by accuracy, and cost as a weighted sum of latency, GPU time, and memory swaps. Also penalize risky events heavily: e.g. an OOM or format error could count as failure (zero reward or a negative penalty). If swapping models incurs delay, subtract a “swap cost” from the reward each time a new model is loaded. Ensure that *higher reward = better user outcome*; e.g. successful correct answers get positive reward, while wasted time or errors get penalized. The exact weightings (λ) depend on your priorities (fast vs. accurate). Keep the reward **explicit and auditable**: log how it is computed, so you can tune it. xRouter emphasizes **explicit cost–performance trade-offs** in the reward. For MoK, one might also include other factors: penalize hallucinations or unsafe outputs (e.g. -∞ if safety check fails), or include metrics for format correctness (reward 0 for improperly formatted answers).

## Benchmark Suite Design  
The evaluation suite must reflect MoK’s real workloads, not just generic NLP tasks. Use a **task mix** that mirrors expected queries: e.g. coding puzzles (like HumanEval), math word problems (GSM8K), open-domain QA (MMLU, GPQA), conversation (MT-Bench), etc. In one routing example, queries were classified into categories and mapped to benchmarks accordingly (code queries to HumanEval, math to GSM8K, general reasoning to MMLU). Similarly, MoK’s benchmarks should weight tasks by their occurrence in production. If your system mostly serves customer support, include email and QA tasks; if it’s a coding assistant, include competitive programming tasks. Avoid only “toy” benchmarks – collect a sample of real user prompts (anonymized) to use as test queries. You might also design adversarial tasks (e.g. tricky prompts) to stress-test routing. In short: assemble a *representative* mix of tasks that the MoK will see, rather than random public benchmarks, so the router is optimized for your domain.

## Human Evaluation Protocol  
Automated metrics only go so far, so design a clear human-eval process for final validation. Use automated checks (unit tests for code, format validators, reference comparisons) for routine correctness. But for nuance and safety, enlist human reviewers. As LabelStudio notes, automated metrics are deterministic and scalable, but **miss subjective quality and edge cases**. A typical protocol: use automated metrics for every candidate (to catch regressions and obvious failures), and **sample outputs for human judgment on a slice** (e.g. new features, high-risk queries). Humans should rate criteria like *helpfulness, factuality, coherence, and adherence to policy*. The guidelines suggest using humans sparingly but strategically: e.g. during canary releases or on “critical slices” (sensitive domains). When automated and human signals disagree, investigate rather than assume one is wrong. In MoK’s case, whenever a new routing strategy or expert is introduced, have a human cohort label a random sample of outputs for correctness and quality, especially in complex or safety-critical tasks (where hallucinations matter). Use a rubric (e.g. scale 1–5 for each dimension) and ensure multiple raters to guard against bias. In summary, **automated evaluations run continuously; human reviews happen periodically on selected examples** to capture what metrics miss.

## Security and Trust Boundaries  
Safety is paramount when orchestration can call arbitrary tools or LLMs.  First, **output sanitization**: never trust raw expert outputs as code or actions without validation. SHIELDA warns that unsanitized LLM outputs (e.g. raw shell commands, SQL) can introduce vulnerabilities like XSS or remote code execution.  Therefore, filter or escape any special tokens, validate JSON schemas, and run untrusted outputs only in sandboxed environments.  Implement **prompt injection defenses**: strip or neutralize embedded instructions in retrieved text or user input. For example, LLMs can inadvertently output injection-prone strings, and malicious content in RAG documents can hijack agents. Thus, apply strict content filtering (e.g. OWASP’s LLM Top-10) and tag or exclude unverified knowledge sources. Tools should run with least privilege: for instance, a code-generation expert should not have direct shell access.  Use separate execution contexts for each expert to prevent one expert’s output poisoning another’s prompt. Finally, maintain **trust scores** for information: e.g., score external docs by source reliability and ignore low-trust ones. In essence, enforce all standard security best practices (input validation, access control, auditing) on the MoK pipeline, because an orchestrator is only as safe as its inputs and outputs.

## Cost Accounting  
Accurate cost measurement is needed to train routers and monitor budgets.  Track **compute cost**: seconds of GPU time for each model call, memory usage, and any disk swap overhead.  Include **latency** as a cost metric (in ms) since slow responses hurt UX.  For local GPUs, measure context-switch overhead: as [9] notes, switching a model often takes 30–60 seconds to load; that overhead should be amortized or penalized.  Also count **cloud/API costs** if using remote LLMs (billing per token) and **development costs** (e.g. fine-tuning GPU-hours). The xRouter system explicitly performs “auditable cost accounting across model calls” – MoK should too.  In practice, tag each log with the actual resource usage and assign monetary cost if relevant.  Maintain a running tally of cost per request (sum of all model-call costs and overhead) so the router can observe the trade-off.  Finally, for evaluation, cost accounting includes any **oracle/reference runs** (e.g. the cost of computing a ground-truth answer if used for scoring). By comprehensively measuring GPU time, memory, I/O, and API charges, MoK can optimize truly cost-aware routing.

## Expert Health and Degradation Tracking  
Experts may degrade over time or behave erratically. Monitor each expert’s **health** continuously. Key signals: recent error rates (timeout, OOM, format errors), and drift in accuracy or latency on standard tasks.  Maintain a running reliability score (as in Section 6 above). For example, update a Bayesian “agent reliability” weight based on each success or failure.  If an expert’s error rate spikes or latency dramatically increases, flag it as unhealthy and temporarily avoid it. Periodically run **heartbeat tests**: send a set of canned queries to each expert and check performance.  Compare against a baseline: if an expert falls below an acceptable accuracy or is significantly slower than normal, trigger an alert.  In essence, treat each expert like a monitored service: keep metrics (throughput, response quality) and a dashboard. If an expert is consistently underperforming (low reward), it should be downgraded or retired. Finally, track **resource health**: if a model is loaded but GPU memory is saturated, that expert is effectively unavailable until freed. In sum, build automated monitors to detect technical availability vs. performance availability issues.

## Adapter Lifecycle Management  
Adapters (fine-tuned models or prompt templates) consume resources and should be managed carefully.  If memory is limited (e.g. an 8GB GPU), only the most critical adapter can stay loaded; others must be loaded on demand.  Use a caching policy: keep the N most-used adapters in memory and evict least-recently-used ones.  Prefetching can help: predict upcoming queries’ domains and pre-load the corresponding adapter ahead of time. For instance, if user switches topic to code, load the code-specialist adapter during the previous turn.  Conversely, if an adapter is rarely used for a long time, unload it to free VRAM.  As [9] notes, GPUs often hold only one large model at a time, and switching costs 30–60s, so reducing swaps is critical.  Also have a “retirement” policy: if a new adapter supersedes an old one (or an expert is deprecated), retire it from the registry and delete its files.  Ultimately, manage adapters like a cache: resident ones should be those that maximize the expected benefit given limited capacity.

## Heterogeneous Hardware Strategy  
MoK should adapt to the hardware profile.  On a **larger GPU (e.g. 24GB)**, you might load bigger models or multiple experts simultaneously. You could even deploy a mid-size model as a lightweight router (a “BERT w/ context” approach).  On **multiple GPUs**, distribute experts: pin half the models on GPU0 and the rest on GPU1, and route queries accordingly.  CPU offload (LLaMA-2 supports it) can allow running >1 model by moving attention layers to CPU, at cost of speed.  In **mixed local/remote** scenarios, treat remote APIs (OpenAI, etc.) as part of your expert pool. Routers can learn to offload certain queries to cloud models; xRouter’s experiments showed routers adaptively favoring certain remote models depending on the input.  The scheduling policy should consider hardware costs: e.g. if two GPUs are available, run two experts in parallel to cut latency, or replicate a large model on both GPUs for redundancy.  As hardware changes, MoK’s design should allow re-training the router: e.g. the reward/cost balance changes if GPU time is cheaper.  In summary, build MoK to detect available resources and adjust its routing logic: more parallelism on beefy machines, more caching on smaller ones, and flexible use of cloud endpoints.

## Observability and Debugging UX  
To diagnose bad routing decisions, MoK needs rich telemetry.  Log every request with its full context: input, chosen expert(s), scores/feature values at router time, outputs, latencies, and outcome.  Collect aggregates: distribution of routes taken, average response time per expert, error frequencies, etc.  Dashboards should show **per-request routing traces**. For example, the LocalForge prototype included a dashboard with routing decisions and memory stats in real time.  MoK should similarly allow visualizing the decision path: e.g. highlight which signal was decisive (maybe by showing the weighted score contributions for each expert).  Include debugging hooks: ability to replay a request through an alternate router or to “force-call” a specific expert and compare.  Also monitor infrastructure: GPU/CPU utilization, VRAM pressure (to catch memory leaks), and queue lengths. Per-request logging should include internal confidence values so that a routing failure can be traced to “it picked model X with low confidence, should have escalated.” In summary, build observability akin to a microservice’s logging: **request metrics, expert performance, routing feature values**, and use them in dashboards/alerts so engineers can pinpoint issues.

## Data Governance  
All conversation and route data must be handled with privacy and security in mind.  Treat user traces as sensitive logs: store them encrypted, with access controls. Before persisting a trace, **redact any PII** (names, emails, credit cards) as recommended by data privacy tools. For example, an AI gateway can automatically replace detected sensitive entities with placeholders or block logs that contain too much private data. Keep data versioned: if you use a trace for training, note which router schema version produced it. Ensure logs are anonymized and stored in a secure database (or vault) with audit trails. If private knowledge (e.g. company docs) is used, sanitize or hash any private text. Finally, comply with regulations: e.g. if GDPR applies, give users the ability to delete their data from the training logs. In short, MoK’s data (both inputs and generated outputs) should be stored as tightly controlled logs, with PII filtering policies in place and clear lifecycles (retention limits, versioning, and access logs).

## Domain Onboarding Workflow  
Adding a new expert for a domain should be a streamlined process.  One recipe: **(1)** Define the domain/task and gather representative data or queries. **(2)** Develop or fine-tune an expert model (or adapter) on that data. **(3)** Create automated tests for the new expert on known benchmarks or examples. **(4)** Add it to the expert registry with metadata (role, capability, context limits). **(5)** Update the router training data: for a rule-based router, add new rules; for a learned router, include labeled examples or logs showing this expert should handle relevant queries. **(6)** Deploy the expert in a sandbox and run canary traffic to verify performance. **(7)** Monitor its behavior and incorporate feedback. Automate as much as possible (e.g. CI that runs evaluation when a new expert is proposed).  For example, if you onboard a “medical summarization” expert, first fine-tune on medical texts, test on medical QA benchmarks, then add it and tag relevant user intents in the router’s classifier. The goal is a repeatable pipeline from “new model trained” to “goes live in MoK”, with clear quality gates at each step.

## Vision Integration Architecture  
If vision capabilities are needed and kept separate from MoK, define a clean handoff. One approach: **preprocess images into text or features**. For example, pass an image to a dedicated vision model (or CLIP) to generate a caption or embedding, then include that text snippet in the prompt along with a `{"image_caption": "..."} ` field. The router then sees a textual representation of the image.  Clearly label it (a “vision flag”) so experts know an image was present. Alternatively, route via a vision expert first: the router might detect an image modality flag and send the request to a vision-specialist that returns text, then feed that into the normal pipeline.  In any case, define the contract: either the LLM experts accept an `image_url` or base64 and call vision tools internally, or require a two-step “image → caption → MoK”. For MoK stability, the simpler “vision-as-text” contract (image → caption) keeps vision logic encapsulated. Cite nothing; just ensure experts expect the same format (e.g. an `!caption()` function).

## Rollout Strategy  
Deploy changes cautiously. For **new routers or policies**, use *shadow mode* first: run the new router in parallel (without affecting live traffic) and compare its decisions to the current one. If its decisions are safe and promising, move to *canary deployment*: route a small percentage of real users (say 5–10%) through the new router. Employ *gated slices*: only expose the new version to low-risk domains or non-critical users initially. Monitor metrics (success rate, latency, user satisfaction) closely. Implement **fallback triggers**: e.g. if the new router yields >5% failure rate increase, automatically revert those sessions to the old system. The labelstudio suggestion applies: **automated metrics for always-on monitoring, human checks before full rollout**. Similarly, for new experts or adapters, start in a supervised mode or limited domain slice. For memory policies, gradually extend across sessions. Each change should be incremental, with clear rollback criteria. Document each release’s impact in logs so that if an issue is found, you can quickly roll back.

## Kill Criteria  
Plan objective criteria for abandoning components if they don’t work. Examples: if a learned router never beats a simple baseline (like “always use Model X”) on a validation set, it should be scrapped. If a new expert domain has persistently low accuracy or causes too many errors in testing, it should be retired. Watch resource use: if a model consistently OOMs or trips the cost budget beyond benefit, drop it. Also consider ROI: if adding an ensemble of experts only improves correctness by <ε% but triples latency, it’s not worth it. Define thresholds (e.g. “router must improve accuracy by ≥2%” or “cost ≤10% overhead”) and measure them. Finally, stay wary of “ideology” bias – if experiments show that a fancy multi-expert scheme isn’t helping in realistic conditions, be ready to revert to simpler designs. In essence, kill components that fail real-world validation or whose cost far outweighs their gains.
